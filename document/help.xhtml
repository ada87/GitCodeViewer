Workspace Navigation Help

The mobile reader supports local browsing, syntax highlighting, and a small set of affordances that change with viewport size. The title in the header is intentionally more interactive on phones than on tablets.

When to use the title link

  • Return to the workspace root without reopening the list page.
  • Jump from a nested file path back to the repository overview.
  • Expose a lightweight affordance on narrow screens where the navigation bar is compact.

Visual rules

Viewport Title treatment Reason
Phone Use a softer accent and a thin underline. The touch target needs a visible hint without overpowering the header.
Tablet Keep the title neutral. The layout already has enough room and the interaction is less ambiguous.
Desktop preview Match the tablet treatment unless there is another navigation pattern. Consistency matters more than emphasis on large viewports.

Troubleshooting

  1. If the title appears left-shifted on a phone, inspect the header flex container and any reserved gutter width.
  2. If a file cannot fold, hide the fold gutter instead of leaving empty space.
  3. If a new suffix renders as plain text, confirm the extension map and dev sample list were updated together.
  4. If a sample looks fake, replace it with a build file, config, guide, or diagram before taking screenshots.

Common questions

Why not underline the title on tablet? The extra space already signals hierarchy, so an interaction hint is less necessary.

Why hide the fold gutter? Empty reserved space is more distracting than a missing fold icon when the mode cannot support folding.

Why maintain realistic sample files? Reviewers judge typography and density from sample content, so placeholder text gives misleading feedback.

Quick gestures

  • Tap the header title to return to the workspace overview.
  • Long press a file to open context actions.
  • Swipe back from the left edge to close the current file.
